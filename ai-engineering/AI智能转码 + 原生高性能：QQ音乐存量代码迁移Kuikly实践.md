## 一、背景

Kuikly 是腾讯开源的新一代高性能跨端开发框架，支持基于 Kotlin 开发 Android、iOS、鸿蒙、Web、小程序及 Mac 多端应用。QQ 音乐已累计开发 200+ Kuikly 页面，但项目中仍遗留较多基于 React 的 H5 页面，团队计划统一改造。

相较于 React H5，Kuikly 原生跨端架构具备天生高性能渲染、低延迟加载优势，可从底层解决 H5 页面加载慢、交互卡顿等痛点。

为了破局，团队联合 Kuikly 团队将 Kuikly 核心知识库"喂"给 AI，构建了一套自动化的 AI 转码工作流。目前已顺利完成 9 个 H5 页面及 3 个公共组件的 Kuikly 改造，AI 代码采用率高达 90%+，中等复杂度页面开发周期从 3 天以上缩短至 1 天以内，页面加载耗时降低 90%。

## 二、转码架构与流程

### 整体架构

采用插槽式设计实现流程逻辑与业务知识解耦，通过更换底层"业务 Skill"，其他业务即可快速接入。

### 转换流程（4 步）

1. **初始化预处理** — 脚本处理 base64/SVG 等内联资源（替换为本地引用避免 token 浪费），缓存检查避免重做
2. **关键字与知识收集** — 扫描源码提取关键字，按需加载框架文档（而非全量注入，解决"注意力稀释"问题）
3. **渐进式代码生成** — 先分析结构 → 再逐文件转换 → 最后架构整理
4. **验证与交付** — 单文件逻辑检查 + 编译检查（循环修正直到通过）

### 目录结构

```
<项目根目录>/
├── react2kuikly/
│   └── [页面名称]/
│       └── cache/
│           ├── progress.md           # 进度跟踪（断点续转）
│           ├── project-keywords.md   # 业务关键字集合
│           ├── knowledge-cache.md    # 统一转化知识库
│           └── conversion-plan.md    # 转换计划与代码分析报告
└── $OUTPUT_DIR/
    ├── [PageName]Pager.kt
    ├── composedview/
    ├── models/
    └── viewmodel/
```

## 三、关键方案演进

### 3.1 智能关键字提取与按需加载

早期全量注入文档导致 Token 膨胀和 AI"注意力稀释"。改进为精准扫描 + 动态组装：基础框架知识全量加载，组件文档和业务规则严格执行按需加载。

### 3.2 内联资源预处理

React 项目中的 base64 图片和长 SVG 节点白白占用上下文窗口。前置脚本自动识别并提取所有内联资源，替换为本地文件引用，同时抹平框架差异（如 SVG color → tintColor）。

### 3.3 渐进式转换策略

| | 一次性全量生成 | 渐进式生成（最终采用） |
|---|---|---|
| 核心思想 | 全部代码一起交给 AI | 先分析结构，再逐文件转换，最后整理 |
| 优点 | 流程极简 | Token 可控，逻辑准确，错误恢复成本低 |
| 缺点 | 上下文硬限制，注意力衰减，可≈控性为零 | — |

### 3.4 进度持久化与断点续转

解决 AI Agent "失忆"和"失控"问题。每个页面有独立 `progress.md`，支持两级粒度，出错后从断点恢复而非全量重做。

### 3.5 双重质量保障

- **转换中检查**：单文件逻辑验证 + 业务 Checklist 检查
- **转换后检查**：全局逻辑验证 + 编译检查（循环修正直到通过）

### 3.6 可插拔业务 Skill 设计

```
react2kuikly（核心 Skill）
    └── 调用 → qqmusic-react2kuikly（业务 Skill）
    └── 调用 → 其他业务 Skill（可自行接入）
```

业务 Skill 通过 PATH_CONFIG 声明资源路径，核心脚本动态解析，无需修改核心代码。
