## 一、出发点

目前做 skill 主要有两种方式：

- **人工编写**：受限于个人经验与精力，效率低，质量波动大，测试覆盖不足，团队间容易重复造轮子。
- **对话生成**（OpenClaw / Claude Code）：虽提升编码速度，但本质是非确定性探索过程，缺乏自动化验证闭环与工程化约束，一次只能生成一版，需多轮交互才能稳定。

因此产生了**面向 Harness 设计的技能工厂**的想法。

## 二、产品功能

### 生产模式 — 标准化测试驱动技能生成

**技能定义** → 用户输入想生成的技能、测试问题、API 接口（后续支持挂载 SOP 文档）。

**基线诊断** → 判断是否需要生成：
- 裸模型评估：直接让模型执行任务作为基线对照。
- Skill 匹配分析：召回相似 skill，评估现有能力是否满足需求。
- 失败点和不确定行为 = Skill 需要解决的真实能力缺口。

**测试构建** → 复用用户输入的测试问题。

**多路并发 Skill 生成** → 并行调用 3 种不同策略的 Creator（不同模型/不同 Prompt 模板），相当于一次买三张彩票，只要一路生成高质量代码就算成功，大幅提高首次生成成功率。

**回归迭代** → 测试 + 优化，从格式规范、复用创新、功能可用性、运行稳定性、文档规范几个维度评价打分并优化。

## 三、生态适配

对知流平台进行了适配，可基于知流上的 MCP、HTTP、Dify agent 工具直接生产技能。

## 四、迭代方向

- **统一入口**：封装分析迭代流程，只呈现结果和分析结论。
- **基于 Trace 的技能机会挖掘 → 规模化生产 + 筛选**：参考千问团队的 Trace2Skill 思路，将 Agent 的隐性经验（执行轨迹）转化为显性知识（结构化技能文档），基于日志数据沉淀可复用的技能。
- **功能优化**：强化回归迭代（如 SkillRL 思路让技能库与策略共同进化），融入更多上下文信息（用户信息、历史数据），甚至引入虚拟沙箱环境供 Agent 测试回归。

## 附录：三种模式对比

| 维度 | 模式一：人工编写 | 模式二：对话生成 | 模式三：SkillFactory |
|---|---|---|---|
| 核心逻辑 | 人脑驱动，依赖个人经验 | 对话驱动，依赖 Prompt 与上下文窗口 | 流程驱动，标准化流水线 + TDD |
| 生产效率 | 低（天/周） | 高 | 高 |
| 代码质量 | 波动大，风格不统一 | 随机性强，缺乏异常处理/类型提示 | 标准化高，下限有保证 |
| 测试验证 | 依赖人工，无自动化闭环 | 无自动化闭环 | **自动化闭环，先有测试再生成，自动回归验证** |
| 多方案探索 | 单线程，推翻重来成本高 | 串行迭代，难并行对比 | **多路并行竞优，3 种策略择优录用** |
| 工程化程度 | 格式规范易出错 | 缺乏鲁棒性与安全规范 | 可注入测试逻辑，达到生产级交付 |
| 知识复用 | 通用逻辑重复造轮子 | 对话记录难共享 | 自动识别重复模块，建议公共库化 |
| 核心缺陷 | 效率瓶颈 + 质量黑盒 | 不可控 + 缺乏工程验证 | 初期搭建成本高 |
