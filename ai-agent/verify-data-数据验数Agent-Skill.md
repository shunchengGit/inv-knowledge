# verify-data：一个端到端的数据验数 Agent Skill

> 作者：晓莄 | 来源：阿里云开发者

## 摘要

本文介绍了一个面向数据开发团队的端到端数据验证 Agent Skill——verify-data。该技能通过自然语言交互，自动完成从表结构获取、基准表发现、代码逻辑分析、验数 SQL 生成、执行到报告发布的全流程，将传统手工验数从"手写多条 SQL + 人工比对"升级为"一句话触发 + 评审级证据输出"。验数时间从 2-4 小时压缩到 30 分钟以内。

## 关键要点

### 核心架构与能力

- **7-9 步主流程 + 条件子步骤可达 17 步**：收集信息 → 基准表自动发现 → 表结构分析 → 代码分析 → SQL 生成 → 执行 → 报告 → 通知
- **基准表自动发现（两阶段策略）**：
  - 第一阶段：血缘发现候选集
  - 第二阶段：指标/维度精排，评分公式 `score = 血缘亲和度 × 0.5 + 维度重合度 × 0.3 + 指标重合度 × 0.2`
  - Top-1 ≥ 0.7 直接选；多表联合可覆盖则走多基准表策略
- **10 类标准化 SQL 模板**：总量对比、数据质量检测、维度逐项对比、维表分区校验、CUBE 完整性、汇总行一致性、逻辑关系校验、历史趋势对比、关联膨胀检测、日期维度关联校验
- **4 种降级验数策略**：无基准表时自动选择 CUBE 自洽校验 / DWD 重算 / 维表静态校验 / 一致性检查，且降级策略强制追加代码审查 + V 类定量证实 SQL

### 5 种验数场景

| 场景 | 触发条件 | 特殊处理 |
|------|----------|----------|
| S1 新模型上线 | 单研发表，无基准表 | 基准表自动发现 + 降级 |
| S2 迭代验数 | DEV vs PROD 对比 | 强制 Code Diff 分析，8 类风险信号扫描 |
| S3 日常监控 | "最近数据异常" | 趋势对比 |
| S4 业务质疑 | "xx 指标对不对" | 口径验证 |
| S5 口径迁移 | "口径变了" | 变更对比 |

### 4 条红线（强制约束）

1. 禁止跳过 SQL 模板直接手写跑数
2. 禁止靠字段名猜映射——必须读运行态代码 + 查 schema
3. 禁止降级策略仅跑 D 类 SQL 就出报告——必须追加代码审查 + V 类定量证实
4. 必须对所有 JOIN 做膨胀率验证、对所有日期维度做日期关联校验

### 结论判定体系

- **PASS**：所有 SQL 通过，无风险
- **WARNING**：主要项通过，但有外部阻塞或非关键差异
- **FAIL**：关键 SQL 失败或差异无法解释

### 关键设计原则

- **模块化依赖设计**：核心验数逻辑与平台实现解耦，替换元数据服务模块即可适配不同环境
- **14 个 References 文档**：主文件是流程路由骨架，细节按需加载
- **3 个容易被忽略的强制/自动步骤**：基准表适用性预检、维表 CUBE 检测、基准表与研发表逻辑对照

### 踩坑经验

1. **CUBE 汇总行 NULL 不是 Bug**：GROUPING SETS 生成的汇总行维度值为 NULL 是标准行为
2. **浮点精度不要较真**：差异 < 1e-6 视为实质为 0
3. **DWD 重算同构只能自证执行一致**：复制 ADS 代码逻辑的 DWD 重算无法发现 ADS 本身的逻辑错误，必须独立构造 V 类 SQL

### 技术栈

- 计算引擎：MaxCompute（ODPS）
- 数据开发平台：DataWorks
- 协作平台：钉钉文档
- Agent 运行时：支持 Skill 定义的 Agent 框架

### 当前挑战

- 执行效率与交互体验：完整流程 15-30 分钟，多个确认节点交互成本偏高
- 权限与环境依赖：跨项目权限不齐全会中断流程
- 降级策略的置信度：无法保证覆盖所有逻辑缺陷
